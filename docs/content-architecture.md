# 内容与架构说明

## 1. 一句话结论

现在这个项目里，唯一应该直接修改的笔记内容目录是：

- `src/notes`

下面这些都不应该直接手改：

- `public/notes`
- `public/index.html`
- `public/css`
- `public/js`
- `src/search/index.json`

因为它们都是构建流程中的输出或中间产物。

---

## 2. 当前架构的真实职责

### 源目录

`src/` 目前承担了三类职责：

1. 内容源
2. 页面模板源
3. 站点资源源

具体对应如下：

| 目录/文件 | 当前职责 | 是否应手改 |
| --- | --- | --- |
| `src/notes/` | 笔记 Markdown 源文件 | 是 |
| `src/_includes/` | Nunjucks 布局与组件模板 | 是 |
| `src/index.njk` | 首页模板 | 是 |
| `src/notes.md` | 所有笔记页模板 | 是 |
| `src/scss/` | 样式源文件 | 是 |
| `src/js/` | 前端脚本源文件 | 是 |
| `src/search/index.json` | 搜索索引中间产物 | 否 |

### 输出目录

`public/` 是完整的网站发布产物：

| 目录/文件 | 当前职责 | 是否应手改 |
| --- | --- | --- |
| `public/index.html` | 首页输出 | 否 |
| `public/notes/` | 所有笔记 HTML 输出 | 否 |
| `public/css/` | 编译后的样式 | 否 |
| `public/js/` | 复制后的浏览器脚本 | 否 |
| `public/search/` | 最终搜索索引输出 | 否 |

---

## 3. 当前构建链是怎样工作的

当前完整构建命令是：

```bash
npm run build
```

它会按下面顺序运行：

1. `clean`
   删除整个 `public/`
2. `styles:prod`
   把 `src/scss/styles.scss` 编译到 `public/css/styles.css`
3. `preprocess`
   对 `src/notes` 做 front matter、链接、公式、搜索索引处理
4. `eleventy`
   从 `src` 读取模板和内容，生成完整的 `public`

这条链路决定了一个事实：

- `public` 任何文件都只是“构建结果”
- 手工修改 `public` 不会被保留

---

## 4. 为什么你会觉得混乱

这是一个很正常的感受，原因主要有三个：

### 1. `src` 里同时放了内容和站点代码

现在 `src/notes`、`src/scss`、`src/js`、`src/_includes` 都在同一个根目录下。

这在 Eleventy 项目里不是错误，但对“主要任务是写笔记”的用户来说，会让 `src` 看起来像一个“混装仓库”。

### 2. `public/notes` 的命名太像“真的笔记目录”

从语义上看：

- `src/notes` 像源文件
- `public/notes` 也像笔记文件

如果没有明确文档，很容易让人误以为两边都能改。

### 3. 旧架构遗留痕迹还在

虽然现在构建链已经被整理过，但目录命名仍沿用了早期结构：

- 没有把“内容源”和“站点实现”物理拆开
- 没有把“发布目录”改成更明显的名字，比如 `dist/`

---

## 5. 当前阶段的操作规则

在正式重构目录之前，建议统一遵守下面这套规则：

### 笔记修改规则

- 只改 `src/notes/**/*.md`

### 页面与样式修改规则

- 模板改 `src/_includes/`、`src/index.njk`、`src/notes.md`
- 样式改 `src/scss/`
- 前端脚本改 `src/js/`

### 构建与检查规则

- 修改完成后统一运行 `npm run build`

### 输出目录规则

- 把 `public/` 视为可随时删除、可完全重建的缓存/发布目录

---

## 6. 推荐的下一阶段整理方案

我建议不要立刻大迁移现有文件，而是采用“先立规则，再迁目录”的方式。

### 阶段 A：立即执行，低风险

目标：先把职责讲清楚，不改大量路径。

建议：

- 继续保留 `src/notes` 作为笔记源
- 明确文档规定 `public/**` 一律不手改
- 继续使用现有 `build` 流程

这一步已经基本完成。

### 阶段 B：中期整理，最值得做

目标：把“内容源”和“站点实现”物理分离。

建议的新目录：

```text
content/
  notes/

site/
  _includes/
  pages/
  scss/
  js/
  tools/

build/
  scripts/

public/
```

对应职责：

- `content/notes`：只放笔记源
- `site`：只放页面模板、样式、脚本、工具页
- `build/scripts`：只放构建脚本
- `public`：只放生成产物

这会把“写内容”和“改网站”清晰分开。

### 阶段 C：为“便于笔记修改的新功能”做准备

目标：让未来的编辑功能建立在清楚的数据边界上。

建议：

1. 先把笔记源统一为单一入口目录
2. 再设计新的笔记修改功能
3. 功能只读写“内容源”，不碰 `public`

这样后续无论你要做：

- 更易写 front matter
- 自动创建新笔记
- 分类/标签管理
- 笔记预览
- 笔记批量重命名

都会更稳。

---

## 7. 我对后续重构的建议顺序

如果你之后要我正式整理目录，我建议按下面顺序做：

1. 先引入 `content/notes`，把笔记源迁过去
2. 调整 Eleventy 输入配置，让模板继续工作
3. 再把 `src` 拆成 `site` 与 `build`
4. 最后再做“便于笔记修改”的新功能

原因是：

- 先稳住“内容源”边界
- 再做功能，返工最少

---

## 8. 当前的官方规则

从现在开始，这个项目里关于内容修改的默认规则是：

> 想改笔记，就改 `src/notes`。  
> `public/notes` 永远只是生成结果，不是编辑入口。

如果后面你决定让我正式迁移到 `content/notes` 架构，这条规则会进一步升级为：

> 想改笔记，就改 `content/notes`。  
> 网站代码和内容源彻底分离。
