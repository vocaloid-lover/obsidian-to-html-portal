# Integral Lab V1 Scope

## 1. 文档目的

本文档用于明确 Integral Lab（积分实验室）第一版的开发范围、技术边界与阶段目标，避免在开发初期因需求扩张导致架构失控、实现延期或与现有静态笔记网站发生不必要的耦合。

第一版的核心目标不是实现一个完整的符号计算系统，也不是构建一个通用数学游戏平台，而是：

> 在现有静态笔记网站中，落地一个可运行的、可演示的、规则驱动的积分交互实验页面。

---

## 2. 第一版目标

第一版需要完成以下目标：

1. 在现有静态笔记网站中新增一个独立页面，用于承载“积分实验室”。
2. 页面能够展示预设积分题目，并以 MathJax 正确渲染数学公式。
3. 用户可以通过点击可用工具，对当前积分表达式执行有限步的规则化变换。
4. 系统可以记录每一步操作历史，并展示变换前后的表达式。
5. 系统能够对当前状态给出有限的工具推荐，而不是完全自动求解。
6. 第一版应作为后续更大型数学交互项目的基础原型，验证以下关键问题：
   - 工具化求解是否适合网页交互
   - 规则驱动架构是否足够清晰
   - 页面结构是否适合未来扩展
   - 与现有静态网站体系是否兼容

---

## 3. 产品定位

Integral Lab V1 的定位如下：

- 它是一个嵌入现有知识型静态网站中的交互工具页
- 它是一个“积分操作实验室”，而不是完整 CAS
- 它强调“用户调用工具逐步变换”，而不是系统一键出答案
- 它优先服务于“展示思路、验证架构、建立后续扩展基础”

第一版不追求题型覆盖广度，不追求输入自由度，也不追求强自动化能力。

---

## 4. 第一版包含范围

## 4.1 页面范围

第一版包含一个新的页面，例如：

- `/tools/integral-lab/`

该页面将嵌入现有静态网站体系中，继承现有布局、主题和全局样式。

页面至少包含以下区域：

1. 页面标题与说明区
2. 当前题目展示区
3. 当前表达式展示区
4. 可用工具区
5. 操作历史区
6. 基础提示或知识链接区

---

## 4.2 功能范围

第一版必须包含以下功能：

### 4.2.1 预设题目加载
- 系统能够加载至少 3 道预设积分题
- 每道题包含标题、难度、标签、初始表达式、提示信息等基础数据

### 4.2.2 当前表达式展示
- 系统能够展示当前积分表达式
- 使用 MathJax 渲染数学公式

### 4.2.3 可用工具展示
- 系统能够根据当前状态列出当前可用工具
- 工具以按钮或卡片形式展示
- 推荐工具可有高亮或提示

### 4.2.4 工具调用
- 用户点击某个工具后，系统执行对应规则
- 当前表达式发生变化
- 变换过程被记录为一步历史

### 4.2.5 历史记录
- 页面能够展示每一步变换
- 历史记录至少包含：
  - 步骤号
  - 工具名称
  - 操作说明
  - 变换前表达式
  - 变换后表达式

### 4.2.6 重置能力
- 用户可以将当前题目恢复到初始状态

---

## 4.3 数学范围

第一版只处理**基础积分交互**，不引入初等数论模块。

第一版建议支持的工具类别限定为以下几类：

1. 常数提出
2. 线性拆分
3. 基础公式匹配
4. 简单化简
5. 固定型换元

第一版中“换元”不要求实现通用换元搜索，只要求支持少量明确题型上的规则化换元。

---

## 4.4 技术范围

第一版将基于现有静态笔记网站的技术栈完成，不引入新的重量级前端框架。

允许使用的核心技术：

- Eleventy
- Nunjucks
- Sass
- 原生 JavaScript / ES Modules
- MathJax

第一版的交互应在前端本地完成，不依赖后端服务。

---

## 5. 第一版明确不做的内容

为控制范围，第一版**不包含**以下内容：

## 5.1 数学能力方面
- 不做初等数论模块
- 不做多数学分支的统一求解器
- 不做通用符号计算系统
- 不做任意积分题自动求解
- 不做复杂启发式搜索

## 5.2 输入方面
- 不做自由 LaTeX 输入解析
- 不做自然语言数学描述解析
- 不做复杂表达式编辑器
- 不做 OCR 或手写识别

## 5.3 用户扩展方面
- 不做用户自定义规则编辑器
- 不做方法插件市场
- 不做在线保存个人方法库
- 不做账号系统

## 5.4 游戏化方面
- 不做完整关卡系统
- 不做成就系统
- 不做经验值、解锁树、道具系统
- 不做剧情或世界观包装

## 5.5 工程方面
- 不引入 React / Vue / Svelte 等新框架
- 不新增后端服务
- 不重构现有网站整体架构
- 不改造现有搜索、主题切换等核心全站功能

---

## 6. 表达式策略

第一版采用“受限内部表达式结构 + MathJax 展示”的策略。

### 6.1 显示层
- 页面展示使用 LaTeX 字符串
- 公式交由 MathJax 渲染

### 6.2 内部处理层
- 数学表达式采用受限 AST / 结构化对象表示
- 第一版仅支持少量基础节点类型，例如：
  - number
  - symbol
  - add
  - mul
  - pow
  - func
  - integral

### 6.3 输入来源
- 第一版的题目数据以预设形式提供
- 表达式结构由题库直接定义
- 不要求从 LaTeX 自动解析为 AST

---

## 7. 架构原则

第一版开发遵循以下原则：

1. **先跑通闭环，再扩展能力**
   - 先实现题目加载、工具调用、历史记录、MathJax 渲染的完整链路
2. **规则驱动优于复杂智能**
   - 先做明确规则，不追求强 AI 感
3. **模块分层清晰**
   - 页面、状态、渲染、引擎、工具规则分离
4. **不破坏现有网站**
   - 新功能应作为独立页面和独立模块接入
5. **优先保证可维护性**
   - 第一版的数据结构和接口设计应为后续扩展留接口

9. 成功标准

如果第一版完成后满足以下条件，则视为范围内目标达成：

网站中成功新增 Integral Lab 页面

页面能够正常展示至少 3 道预设积分题

页面能够根据当前状态显示至少 3 个可用工具

用户点击工具后，表达式和历史记录会正确更新

页面中数学公式显示正常

页面不会破坏现有网站的构建、样式或主题系统

代码结构清晰，可为后续扩展数论模块、自定义规则、更多题型保留空间

10. 第一版后的可扩展方向

以下内容不属于第一版，但应在架构上为其保留可能性：

初等数论模块

更多积分工具

方法推荐优化

自由输入

用户自定义规则

更强的游戏化包装

更复杂的表达式树与规则系统

11. 一句话总结

Integral Lab V1 是一个嵌入现有静态笔记网站中的、规则驱动的积分交互实验页面；它以少量预设题目和有限数学工具为核心，验证“图形化数学操作 + 工具化求解”这一产品方向是否成立。
